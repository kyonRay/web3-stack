# EIP-721 (ERC-721): Non-Fungible Token Standard

> **Date:** 2024-01-01  
> **Tags:** eip-721, erc-721, nft, non-fungible, ethereum, standard

## Introduction

**EIP-721**, finalized in January 2018 by William Entriken, Dieter Shirley, Jacob Evans, and Nastassia Sachs, defines the Non-Fungible Token (NFT) standard. Unlike ERC-20 where all tokens are identical, each ERC-721 token has a unique `tokenId` and distinct ownership. This property makes ERC-721 ideal for digital art, collectibles, in-game items, real-world asset tokenization, and identity.

---

## 1. Core Interface

```solidity
interface IERC721 {
    // ── Events ────────────────────────────────────────────────────────────────
    event Transfer(address indexed from, address indexed to, uint256 indexed tokenId);
    event Approval(address indexed owner, address indexed approved, uint256 indexed tokenId);
    event ApprovalForAll(address indexed owner, address indexed operator, bool approved);

    // ── Ownership & Balance ───────────────────────────────────────────────────
    function balanceOf(address owner) external view returns (uint256 balance);
    function ownerOf(uint256 tokenId) external view returns (address owner);

    // ── Transfer ──────────────────────────────────────────────────────────────
    function safeTransferFrom(address from, address to, uint256 tokenId, bytes calldata data) external;
    function safeTransferFrom(address from, address to, uint256 tokenId) external;
    function transferFrom(address from, address to, uint256 tokenId) external;

    // ── Approval ──────────────────────────────────────────────────────────────
    function approve(address to, uint256 tokenId) external;
    function setApprovalForAll(address operator, bool approved) external;
    function getApproved(uint256 tokenId) external view returns (address operator);
    function isApprovedForAll(address owner, address operator) external view returns (bool);
}
```

## 2. Transfer Model

ERC-721 has two approval levels:

| Mechanism | Scope | Use Case |
|-----------|-------|----------|
| `approve(spender, tokenId)` | Single token | One-time sale or transfer |
| `setApprovalForAll(operator, true)` | All tokens owned by caller | Marketplace listing, game contracts |

### 2.1 `safeTransferFrom` vs `transferFrom`

`safeTransferFrom` checks that the recipient is either an EOA or a contract implementing `IERC721Receiver`. This prevents tokens from being permanently locked in contracts that cannot handle them.

```solidity
interface IERC721Receiver {
    function onERC721Received(
        address operator,
        address from,
        uint256 tokenId,
        bytes calldata data
    ) external returns (bytes4); // must return this selector: 0x150b7a02
}
```

**Always use `safeTransferFrom` in user-facing code.** Use `transferFrom` only when you have verified the recipient contract handles ERC-721.

## 3. Metadata Extension

The optional metadata interface allows each token to have a name, symbol, and URI:

```solidity
interface IERC721Metadata is IERC721 {
    function name()                             external view returns (string memory);
    function symbol()                           external view returns (string memory);
    function tokenURI(uint256 tokenId)          external view returns (string memory);
}
```

### 3.1 Token URI and Metadata JSON

`tokenURI` typically returns an HTTPS or IPFS URL pointing to a JSON file:

```json
{
  "name": "CryptoPunk #3100",
  "description": "One of 9 Alien CryptoPunks",
  "image": "ipfs://QmXx.../3100.png",
  "attributes": [
    { "trait_type": "Type",       "value": "Alien" },
    { "trait_type": "Accessory",  "value": "Headband" }
  ]
}
```

**Centralized vs Decentralized URIs**

| Storage | Pros | Cons |
|---------|------|------|
| HTTPS server | Fast, updatable | Centralized, can go offline |
| IPFS (content-addressed) | Immutable, decentralized | Pinning required for persistence |
| Arweave | Permanent, decentralized | Higher upfront cost |
| Fully on-chain (base64 SVG) | Truly permanent | Very expensive for complex images |

## 4. Enumerable Extension

```solidity
interface IERC721Enumerable is IERC721 {
    function totalSupply()                              external view returns (uint256);
    function tokenOfOwnerByIndex(address owner, uint256 index) external view returns (uint256 tokenId);
    function tokenByIndex(uint256 index)                external view returns (uint256);
}
```

> ⚠️ Enumerable extension is gas-expensive because it maintains sorted lists in storage. Avoid it for contracts with large supplies; use off-chain indexers (The Graph, Moralis) instead.

## 5. ERC-721A: Batch Minting Optimization

[ERC-721A](https://www.erc721a.org/) by Azuki is a widely-adopted extension that reduces the cost of minting multiple NFTs in one transaction from O(n) to O(1) by lazily computing ownership.

**How it works:** Instead of writing the owner for each token, it writes one record and uses a bitmap to mark which tokenIds share that owner. `ownerOf` walks back through the bitmap to find the most recent ownership record.

```solidity
import "erc721a/contracts/ERC721A.sol";

contract MyNFT is ERC721A {
    function mint(uint256 quantity) external payable {
        _mint(msg.sender, quantity); // single SSTORE regardless of quantity
    }
}
```

## 6. EIP-2981: Royalty Standard

EIP-2981 adds a `royaltyInfo` function that marketplaces can call to determine how much royalty to pay to the creator on secondary sales:

```solidity
interface IERC2981 {
    function royaltyInfo(uint256 tokenId, uint256 salePrice)
        external view returns (address receiver, uint256 royaltyAmount);
}
```

> Note: On-chain royalty enforcement is optional — marketplaces may ignore it. Protocols like Zora and Manifold enforce royalties at the contract level.

## 7. Common Patterns

### 7.1 Reveal Mechanic

To avoid metadata sniping before mint ends:

1. Deploy with a placeholder `baseURI` pointing to a "pre-reveal" image.
2. After mint ends, update `baseURI` to the final IPFS folder.
3. Optional: include a `provenanceHash` — SHA-256 of all final images concatenated — committed on-chain before reveal to prove the ordering was not manipulated.

### 7.2 Soulbound Tokens (EIP-5192)

Soulbound tokens are non-transferable NFTs representing identity, reputation, or credentials. EIP-5192 marks a token as locked:

```solidity
interface IERC5192 {
    event Locked(uint256 tokenId);
    event Unlocked(uint256 tokenId);
    function locked(uint256 tokenId) external view returns (bool);
}
// Override _beforeTokenTransfer to revert on transfer when locked
```

## 8. Security Considerations

| Risk | Description | Mitigation |
|------|-------------|------------|
| Reentrancy via `onERC721Received` | Malicious receiver calls back into contract | Follow CEI, use `nonReentrant` |
| `setApprovalForAll` phishing | Fake marketplace gets operator rights | Verify contract addresses before signing |
| Uninitialized upgradeable proxy | Storage slot 0 clashes with `_owner` | Use `ERC721Upgradeable` initializer |
| Metadata centralization | HTTPS URI can return different content | Prefer IPFS with pinning |
| Token ID enumeration | Sequential IDs reveal supply | Use randomized IDs or obfuscate |
